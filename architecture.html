1. System overview

Cortex is a personal knowledge engine that runs entirely on your machine. It ingests documents, code, URLs, and quick notes, extracts atomic facts via LLM, builds an entity graph, and serves everything through hybrid semantic + keyword search.

Three surfaces connect Cortex to Claude Code:

Hooks (automatic)

Registered in ~/.claude/settings.json. Fire on every prompt and tool use. Zero user action required.

Hot context (passive)

Top 20 relevant facts written to ~/.cortex/CLAUDE.md, auto-imported into every session.

MCP tools (on-demand)

7 tools for deep knowledge access. Claude calls them when it needs more than the injected context.

2. Three-layer knowledge model

Cortex stores knowledge at three levels of abstraction. Different questions need different layers.

Layer What's stored When it wins
Chunks 512-token text blocks with contextual prefixes + 768-dim embeddings Raw section recall: "show me that paragraph"
Facts LLM-extracted atomic statements with category, confidence, importance, temporal validity Structured knowledge: "what do we know about X"
Entity graph Named nodes (people, topics, documents) with typed relations Connections: "what's related to X"

The same ingested document populates all three layers. Chunks give breadth, facts give precision, the graph gives relationships.

3. Installation & setup

Install

npm install -g @anmol-srv/cortex

Initialize

cortex init

Runs an interactive wizard that:

  1. Picks LLM provider (defaults to claude-cli — uses Claude Code subscription, no API key)
  2. Picks embedding provider (Ollama default)
  3. Pulls nomic-embed-text Ollama model if missing (~270MB)
  4. Sets default namespace
  5. Creates ~/.cortex/.env with chosen settings
  6. Initializes PGlite database at ~/.cortex/db and runs migrations
  7. Writes ~/.cortex/CLAUDE.md with Cortex instructions + hot-context block
  8. Adds @~/.cortex/CLAUDE.md to ~/.claude/CLAUDE.md
  9. Registers UserPromptSubmit + PostToolUse hooks in ~/.claude/settings.json

Verify

cortex doctor

Checks every component: config file, database, LLM provider, embedding provider, hook registration, CLAUDE.md integration. Exits with non-zero status if anything fails.

4. Ingestion pipeline

Every piece of content (cortex ingest, cortex remember, MCP ingest tool) flows through six stages.

0

CLASSIFY

LLM classifies the input into one of three routes:

thought — Short personal fact or note. Skip chunking / contextualization, store directly.
knowledge — Document or structured content. Run the full pipeline.
noise — Trivial, meaningless (e.g., "hi"). Discard.

Heuristic fast-paths: empty/tiny → noise; long-form (>2000 chars) → knowledge. LLM only called for ambiguous cases.

1

DEDUP CHECK

Compute SHA-256(content). Compare against the document table. If hash matches, skip without parsing, embedding, or LLM calls.

Why hash before parse? Saves the full pipeline cost on re-ingestion. Running cortex ingest "docs/**/*.md" repeatedly is essentially free.
2

PARSE

Format-specific parser extracts sections and clean text:

markdown.jsSplits by headings, preserves hierarchy
code.jsSplits by functions / classes (JS, TS, Python, Go, Rust, Java, C++, etc.)
html.jsStrips tags, extracts readable text
json-parser.jsConverts JSON to readable prose
text.jsParagraph-based split for plain text
3

CHUNK + CONTEXTUALIZE + EMBED

Sections are split into ~512-token chunks with 50-token overlap to preserve context across boundaries.

Each chunk gets a contextual prefix via LLM:

"This chunk describes the authentication middleware
 in the Express.js backend."

Chunks are then embedded (768-dim Ollama nomic-embed-text or 1536-dim OpenAI) and stored in the chunk table with their pgvector embeddings.

4

EXTRACT FACTS

Each chunk is sent to the LLM with the extraction prompt. The LLM returns atomic facts:

Input: "We migrated from MySQL to PostgreSQL in Q3 2024
because of JSON support and pgvector."

Facts:
 · Database migrated from MySQL to PostgreSQL
   [architecture, high, vital]
 · Migration happened in Q3 2024
   [timeline, high, supplementary]
 · PostgreSQL chosen for JSON + pgvector
   [decision, high, vital]

Each fact has: category · confidence · importance (vital/supplementary) · valid_from / valid_until.

Each fact then runs through AUDM (next section).

5

LINK ENTITIES

Topic entities are extracted from the facts via a separate LLM call. Each entity runs through 3-stage resolution (section 6). Entities are linked to facts via the fact_entity junction table. Relationships between entities are stored in the relation table with temporal validity and mention counts.

5. AUDM deduplication

AUDM = Add / Update / Delete / Merge. For each new fact, Cortex checks similarity against existing facts using pgvector cosine distance, then decides:

sim ≥ 0.88→SKIP — paraphrase of existing fact
sim ≥ 0.65→LLM decides: ADD / UPDATE / DELETE / MERGE
sim < 0.65→ADD — clearly new fact

LLM decisions (from prompts/audm-decision.md):

  • ADD — new information, store alongside the existing fact
  • UPDATE — newer version of the same fact; old fact marked superseded
  • DELETE — new fact contradicts the old one; old fact marked invalid
  • MERGE — combine both into a richer fact

Every decision is logged to the history table — full audit trail of knowledge evolution.

Why this matters. Without AUDM, re-ingesting the same content over time pollutes the knowledge base with duplicates. With AUDM, facts stay clean and current — old versions are superseded, contradictions are flagged.

6. Entity resolution

When a new entity name enters the system, Cortex runs a 3-stage resolution cascade to avoid duplicates.

1

EXACT NAME MATCH

DB lookup on entity.name within the active namespace. Fast, no LLM calls.

2

EMBEDDING SIMILARITY + LLM VERIFY

Embed the name, find top-3 similar entities via pgvector, ask LLM: "Are these the same real-world entity?"

Catches cases like "React" ≡ "React.js" ≡ "ReactJS" without hardcoded rules.

3

CREATE NEW

If no match found, create a new entity with its own embedding and type.

7. Search pipeline

A single call to search(query) orchestrates cognitive routing, hybrid retrieval, and graph enhancement.

User Query: "what database do we use" │ ▼ ┌──────────────────────────────────────────┐ │ COGNITIVE ROUTER │ │ classifies query intent (cached): │ │ preference | factual | entity_lookup │ │ | exploratory | temporal │ │ sets categories, graph, expand, limit │ └──────────────┬───────────────────────────┘ │ ┌─────────┴─────────┐ ▼ ▼ ┌────────────┐ ┌────────────┐ │ VECTOR │ │ KEYWORD │ │ SEARCH │ │ SEARCH │ │ embed → │ │ tsvector + │ │ pgvector │ │ ts_rank │ │ cosine │ │ full-text │ └─────┬──────┘ └──────┬─────┘ │ │ └─────────┬─────────┘ ▼ ┌──────────────────────────────────────────┐ │ RECIPROCAL RANK FUSION │ │ score = Σ weight / (K + rank) │ │ vector weight: 1.0 │ │ keyword weight: 0.7 │ │ K = 20 │ │ vital facts promoted at ties │ └──────────────┬───────────────────────────┘ │ ▼ ┌──────────────────────────────────────────┐ │ GRAPH ENHANCEMENT (optional) │ │ extract entities from results → │ │ traverse entity graph → │ │ fetch facts for related entities → │ │ rerank with related facts │ └──────────────────────────────────────────┘

Search latency is ~30ms on thousands of facts (p50 31ms, p95 61ms).

Query expansion

For exploratory queries, the router triggers expansion:

"what tech stack should I use?"
  → "technology preferences"
  → "preferred programming languages"
  → "frameworks to avoid"
  → "recommended tools and libraries"

Each variant is searched independently; results are merged. Surfaces facts that don't literally match the query.

8. Data model

document ──1:N──► chunk (raw text segments with embeddings) document ──1:N──► fact (atomic knowledge with embeddings) fact ──N:N──► entity (via fact_entity junction) entity ──N:N──► entity (via relation table) fact ──1:N──► history (AUDM audit trail)
Table Purpose Key columns
document Source registry source_path (UNIQUE), content_hash, namespace, title
chunk Raw text segments content, section_heading, contextual_prefix, embedding
fact Atomic knowledge uid, content, category, confidence, importance, embedding, status, valid_from, valid_until, access_count
entity People, topics, documents uid, name, entity_type, embedding, mention_count, merged_with
relation Entity relationships source_id, target_id, relation_type, mention_count, invalid_at
fact_entity Fact ↔ entity junction fact_id, entity_id
history AUDM audit trail fact_id, action, old_content, new_content, timestamp
llm_log LLM call observability provider, model, caller, input_tokens, output_tokens, cost, duration_ms

Indexes

  • HNSW indexes on chunk.embedding and fact.embedding for fast pgvector search
  • tsvector + GIN index on content for keyword search
  • Composite indexes on (namespace, status) for filtered scans
  • Unique constraint on (source_id, target_id, relation_type) in relation

9. LLM provider system

Cortex supports four LLM providers with zero-config auto-detection.

Provider API key Cost Default model
claude-cli None (uses Claude Code subscription) Free with subscription haiku
openai OPENAI_API_KEY $0.15/M input (gpt-4o-mini) gpt-4o-mini
anthropic ANTHROPIC_API_KEY $0.80/M input (Haiku) claude-haiku-4-5-20251001
ollama None (fully local) Free qwen2.5:7b

Auto-detection waterfall

LLM_PROVIDER set→use it
ANTHROPIC_API_KEY→anthropic
OPENAI_API_KEY→openai
Ollama reachable→ollama
claude in PATH→claude-cli
none above→actionable error

Per-task provider:model overrides

# Use cheap claude-cli for extraction
LLM_EXTRACTION_MODEL=claude-cli:haiku

# Use Anthropic API for accurate AUDM decisions
LLM_DECISION_MODEL=anthropic:claude-sonnet-4-6

# Use Ollama for entity matching (no API cost)
LLM_ENTITY_MODEL=ollama:qwen2.5:7b

10. Claude Code integration

Three complementary mechanisms make Cortex invisible to Claude while always being used.

Hooks (automatic)

cortex init writes two hook entries to ~/.claude/settings.json:

{
  "hooks": {
    "UserPromptSubmit": [{
      "hooks": [{
        "type": "command",
        "command": "node <cortex>/hooks/user-prompt-submit.js",
        "timeout": 10,
        "statusMessage": "Searching memory..."
      }]
    }],
    "PostToolUse": [{
      "matcher": "Edit|Write|Bash",
      "hooks": [{
        "type": "command",
        "command": "node <cortex>/hooks/post-tool-use.js",
        "timeout": 10,
        "async": true
      }]
    }]
  }
}

UserPromptSubmit flow

  1. Claude Code pipes JSON to hook stdin
  2. Hook reads prompt, calls search()
  3. Returns additionalContext with top matching facts
  4. Claude sees the context automatically — no ! cortex search needed

PostToolUse flow

  1. After every Edit/Write/Bash tool call, hook fires async (non-blocking)
  2. Filters out noisy commands (ls, cat, grep, npm test, node invocations, etc.)
  3. Captures meaningful observations ("Edited src/auth.js", "Ran git push")
  4. Stores directly as a fact in the observation category — no LLM call, fast

Hot context (passive)

Top 20 facts (scored by importance × access_count × recency) are written to ~/.cortex/CLAUDE.md, auto-imported via @import into every Claude session. Refreshed after cortex remember and cortex ingest, or manually via cortex context.

MCP tools (on-demand)

When Claude needs more than the injected context, it calls MCP tools directly.
